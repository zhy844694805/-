# Flutter 待办应用月视图 MVP 设计文档

## 1. 背景与目标

本项目要从零创建一个基于 Flutter 的待办任务应用，产品方向对标滴答清单，但第一阶段只聚焦手机端 MVP，并把月视图体验作为核心竞争点。

MVP 的核心目标是：

- 提供 `iOS/Android` 手机端应用
- 支持邮箱验证码链接（Magic Link）登录
- 支持云同步
- 支持离线可用，联网后自动同步
- 以 `月历 + 当日详情` 作为主工作台
- 支持基础待办能力：标题、截止日期、完成状态、所属清单

## 2. 已确认范围

### 2.1 目标范围

- 平台：`iOS/Android 手机端`
- 账号：`邮箱验证码链接登录`
- 后端：`Firebase Auth + Cloud Firestore`
- 核心页面：`登录页`、`月视图主页`、`任务编辑页`
- 核心交互：点击某一天，在底部详情区查看和勾选任务
- 首版任务字段：标题、截止日期、完成状态、所属清单
- 首版体验重点：月视图体验优先，其他功能够用即可

### 2.2 非目标范围

以下能力明确不进入第一阶段 MVP：

- 提醒
- 重复任务
- 标签
- 优先级
- 拖拽改期
- 周视图、日视图、时间轴
- 多日事件条
- 第三方登录（Apple / Google）
- 桌面端 / Web 适配
- 复杂冲突解决策略
- 独立同步状态中心页面

## 3. 产品成功标准

当用户完成登录后，应能在一个连续闭环内完成以下操作：

1. 进入月视图主页，默认看到当前月份
2. 在月历格中快速识别哪些日期有任务
3. 点击某一天后，在底部详情区查看当天任务
4. 在详情区直接勾选任务完成
5. 新建任务时默认继承当前选中日期
6. 离线时仍能查看已加载月份和任务，并允许新增、编辑、勾选
7. 恢复联网后自动同步到云端

如果这条闭环顺畅，首版目标即视为达成。

## 4. 用户体验设计

### 4.1 主页面结构

主页面采用固定的 `月历 + 当日详情` 结构：

- 顶部区域：清单筛选、回到今天、月份切换
- 中部区域：7 列月历网格
- 底部区域：当前选中日期的任务详情区
- 浮动按钮：新增任务

该结构的目标是同时满足“按月浏览”和“当天执行”。

### 4.2 月视图展示规则

- 月视图使用固定 `6` 行网格，补齐前后月份日期
- 每个日期格最多直接显示 `2` 条任务摘要
- 超过 `2` 条时，以 `+N 项` 表示
- 当前选中日期使用高亮底色
- 今天日期额外有“今天态”强调
- 无任务日期保持轻量显示，避免视觉噪音
- 非当前月份日期以弱化样式展示
- 月格中的任务摘要只承担“存在感”与“快速识别”职责，不承担完整编辑职责

### 4.3 核心交互

- 首次进入月视图主页时，默认选中今天
- 点击日期：切换当前选中日，并刷新底部详情区
- 点击相邻月份日期：切换到对应月份，并自动选中该日期
- 点击月份切换按钮：切换到上一月或下一月，优先保留当前“日”数字；若目标月不存在该日期，则自动取该月最后一天
- 点击任务勾选框：直接完成或取消完成
- 点击任务行：进入任务编辑页
- 点击新增按钮：进入新增任务页，默认带入当前选中日期
- 点击回到今天：跳转到当前月份并选中今天

### 4.4 首版限制

- 月格不支持拖拽任务改期
- 月格不展示完整长文本任务
- 月格不支持跨天视觉连接
- 底部详情区不做复杂折叠手势，默认固定展示

### 4.5 认证与进入应用流程

- 未登录用户打开应用时，进入登录页
- 用户输入邮箱后，应用请求发送 Magic Link
- 用户点击邮件中的登录链接后，通过移动端深链返回应用
- 应用完成 Firebase Auth 登录确认后，进入月视图主页
- 已登录用户再次打开应用时，直接恢复到月视图主页

## 5. 系统架构

应用架构分为 4 层：

### 5.1 展示层

负责页面、组件、动画和手势，包括：

- 登录页
- 月视图主页
- 任务编辑页
- 月历网格组件
- 当日详情区组件
- 任务项组件

### 5.2 状态与用例层

负责承接 UI 动作并组织业务操作，包括：

- 当前登录状态管理
- 当前月份状态
- 当前选中日期状态
- 当前月份任务流
- 当前日期详情任务流
- 新建 / 编辑 / 勾选任务用例

这一层隔离页面和数据源，避免 UI 直接操作 Firebase。

### 5.3 领域模型层

首版只定义最小模型：

- `Task`
- `TaskList`
- `CalendarDaySummary`

不引入提醒、重复、优先级等扩展对象，保证月视图和数据流足够清晰。

### 5.4 数据层

数据层包含：

- `Firebase Auth` 认证接入
- `Cloud Firestore` 任务与清单数据存储
- Repository 接口与实现
- Firestore DTO 与领域模型转换

## 6. 数据模型设计

### 6.1 用户

路径：`users/{userId}`

建议字段：

- `email`
- `defaultListId`
- `lastViewedMonth`
- `createdAt`
- `updatedAt`

### 6.2 清单

路径：`users/{userId}/lists/{listId}`

建议字段：

- `name`
- `color`
- `sortOrder`
- `createdAt`
- `updatedAt`

### 6.3 任务

路径：`users/{userId}/tasks/{taskId}`

建议字段：

- `title`
- `listId`
- `dueDateKey`
- `isCompleted`
- `completedAt`
- `createdAt`
- `updatedAt`

### 6.4 日期处理规则

- `dueDateKey` 保存为用户本地日历日期字符串，格式固定为 `YYYY-MM-DD`
- 月视图查询、分组和展示均以 `dueDateKey` 为准
- 首版编辑页只编辑“日期”，不编辑具体时间
- 若未来新增提醒或具体时间，再增加独立的时间字段，而不修改现有月视图主逻辑

这样可避免首版因为时区、夏令时和自然日边界导致任务归属错误，同时保留后续扩展空间。

## 7. 同步与离线策略

### 7.1 方案选择

首版直接依赖 `Cloud Firestore` 在移动端提供的离线持久化能力，不单独实现本地数据库与自定义同步引擎。

### 7.2 读取策略

- 登录成功后，拉取当前用户的清单
- 进入月视图后，按 `dueDateKey` 查询“当前月份窗口内”的任务
- 页面在内存中按天聚合为 `CalendarDaySummary`
- 切换月份时切换查询窗口

### 7.3 写入策略

- 新建任务：先写本地缓存，再由 Firestore 同步到远端
- 编辑任务：同上
- 勾选完成：同上

对用户而言，离线与在线交互路径保持一致。

### 7.4 冲突策略

首版采用 `最后写入覆盖` 策略。

原因：

- MVP 只面向单账号个人任务管理
- 与完整协作型冲突解决相比，成本更低
- 足以满足首版目标

### 7.5 离线边界

首版可保证：

- 已加载月份可离线查看
- 已加载任务可离线编辑
- 离线可新增任务
- 恢复联网后自动同步

首版不保证：

- 从未打开过的月份在离线状态下仍能完整加载
- 用户可查看细粒度同步队列

## 8. 工程结构建议

建议的目录结构如下：

```text
lib/
  app/
  core/
  data/
  features/
    auth/
    calendar/
    tasks/
```

各目录职责：

- `app/`：应用入口、主题、路由
- `core/`：通用组件、日期工具、错误模型、Firebase 初始化
- `data/`：DTO、数据源、repository 实现
- `features/auth/`：登录与深链处理
- `features/calendar/`：月视图、日期选择、月份切换、聚合逻辑
- `features/tasks/`：详情区、任务编辑、任务操作

该结构确保月视图功能能独立演进，而不会被认证或任务表单逻辑污染。

## 9. 技术选型

- UI 框架：`Flutter`
- 认证：`Firebase Auth`
- 数据库：`Cloud Firestore`
- 状态管理：`Riverpod`
- 路由：`go_router`
- 日期与本地化：`intl`
- 必要时区处理：轻量封装，必要时引入 `timezone`

不推荐首版引入重型日历组件。月视图应自定义实现，以保证：

- 日期格信息密度可控
- 当日详情区交互可控
- 视觉与行为可持续贴近产品目标

## 10. 错误处理

首版需要明确处理以下错误场景：

- Magic Link 登录失败或链接过期
- 网络断开
- Firestore 写入失败
- 数据解析异常
- 页面首次加载为空状态

处理原则：

- 认证失败：给出明确错误提示，并允许重新发送登录链接
- 离线状态：不阻断本地操作，用轻提示说明“等待同步”
- 写入失败：在本地状态上可见，并允许用户重试
- 空数据：提供清晰空态，引导用户创建首条任务

首版不设计复杂错误中心，只在局部页面和交互点提供可理解反馈。

## 11. 测试策略

### 11.1 单元与仓储测试

重点验证：

- Firestore DTO 与领域模型互转
- 日期聚合逻辑
- 月份窗口查询逻辑
- 勾选完成后的状态更新逻辑

### 11.2 Widget 测试

重点验证：

- 点击某一天后，底部详情区切换到对应日期
- 点击任务勾选框后，任务状态发生变化
- 月格摘要与详情区状态保持一致
- 新增任务时默认带入当前选中日期

### 11.3 集成测试

至少覆盖一条关键用户路径：

- 登录
- 进入月视图
- 选择某天
- 新建任务
- 勾选完成

测试重点是行为闭环，不是像素级 UI 还原。

## 12. 风险与后续演进

首版的主要风险：

- 自定义月视图需要较细致的布局与状态同步处理
- 本地时区和日期边界如果处理不统一，容易导致任务归属错误
- 离线体验虽由 Firestore 支持，但 UI 层仍需清楚表达“本地已改、远端待同步”

第二阶段可以在此设计上增量扩展：

- 提醒
- 重复任务
- 标签与筛选
- 拖拽改期
- 更完整的本地缓存策略

## 13. 结论

本项目第一阶段采用：

- `Flutter`
- `Firebase Auth`
- `Cloud Firestore`
- `自定义月视图`
- `月历 + 当日详情` 主工作台

这是在“月视图体验优先”的前提下，实现速度、离线能力和后续可扩展性之间最平衡的方案。
