# 微信小程序双人待办月视图 MVP 设计文档

## 1. 背景与目标

本项目要做一个以 `月视图 + 当日任务详情` 为核心工作台的待办应用，首发与唯一客户端为 `微信小程序`。

本次调整后的产品目标是：

- 首发平台为 `微信小程序`
- 登录方式使用 `微信身份体系`
- 后端采用 `微信云开发`
- 保持 `月视图 + 当日详情` 为主页面
- 支持 `双人空间`
- 支持 `共享清单 + 个人清单`
- 支持任务指派给 `我 / 对方 / 双方`
- 支持强离线：断网可查看、勾选、新增、编辑，恢复后自动同步

首版的重点不是做复杂协作平台，而是先把“双人一起规划生活/待办”的核心闭环做顺。

## 2. 已确认范围

### 2.1 目标范围

- 平台：`微信小程序`
- 登录：`微信身份登录`
- 后端：`微信云开发`
- 数据能力：云数据库 + 云函数
- 首页结构：`月视图 + 当日任务详情`
- 用户模式：支持单人先使用，后续再邀请另一人加入
- 空间模型：一个账号最多加入 `1` 个双人空间
- 数据可见性：同时支持 `共享清单` 和 `个人清单`
- 首页默认视角：`全部`
- 任务字段：标题、日期、完成状态、所属清单、指派对象
- 强离线：断网下继续使用已缓存数据，并允许本地写入后续同步

### 2.2 非目标范围

以下能力明确不进入第一阶段 MVP：

- 提醒
- 微信订阅消息通知
- 重复任务
- 标签
- 优先级
- 多空间切换
- 多人群组协作
- 评论、动态、操作日志
- 复杂清单排序和归档系统
- 周视图、日视图、时间轴
- 拖拽改期
- Web / 原生 App 多端支持
- 复杂冲突解决 UI

## 3. 产品成功标准

用户应能顺畅完成以下闭环：

1. 使用微信身份进入小程序
2. 未绑定双人空间时，也可先创建个人清单和个人任务
3. 在首页月视图中快速识别哪些日期有任务
4. 点击某一天后，在底部详情区查看当天任务
5. 在详情区直接勾选任务完成
6. 新建任务时默认继承当前选中日期
7. 发起邀请并成功与另一位用户绑定为双人空间
8. 绑定后创建共享清单，并在共享清单下创建共享任务
9. 支持任务指派给我、对方或双方
10. 离线状态下继续查看、勾选、新增、编辑，恢复网络后自动同步

只要这条闭环稳定、清晰，首版目标即达成。

## 4. 用户体验设计

### 4.1 首页结构

首页采用固定的 `月视图 + 当日详情` 结构：

- 顶部区域：月份切换、回到今天、范围筛选（`全部 / 共享 / 我的`）
- 中部区域：固定 `6 x 7` 月历网格
- 底部区域：当前选中日期的任务详情
- 浮动按钮：新增任务

该结构的目标是把“按月规划”和“当天执行”放在同一页面内完成，减少小程序多层跳转。

### 4.2 首页默认视角

- 首页默认展示 `全部`
- `全部` 展示：
  - 共享清单任务
  - 当前用户自己的个人清单任务
- 不展示对方的个人清单任务
- `共享` 只显示共享清单任务
- `我的` 只显示当前用户自己的个人清单任务

### 4.3 月视图展示规则

- 使用固定 `6` 行网格，补齐前后月份日期
- 每个日期格最多直接展示少量任务提示
- 超出时显示聚合提示，如 `+N`
- 当前选中日期高亮
- 今天额外有“今天态”强调
- 非当前月份日期弱化显示
- 月格只承担“发现任务分布”的职责，不承担完整编辑职责

### 4.4 当日详情区

- 点击日期后刷新当天任务列表
- 任务项直接展示：
  - 标题
  - 所属清单
  - 指派对象
  - 完成状态
- 支持直接勾选完成
- 点击任务行进入编辑页

### 4.5 新用户与邀请流程

- 用户首次进入小程序时，直接通过微信身份进入
- 如果尚未加入任何双人空间，允许先单人使用
- 首页提供明确入口：`邀请搭子一起用`
- 发起邀请后，通过微信分享把邀请卡片发给对方
- 对方从分享卡片进入后完成绑定
- 绑定成功后回到首页，开始使用共享清单能力

### 4.6 首版限制

- 不做提醒
- 不做复杂 onboarding
- 不做多空间
- 不做复杂清单管理页面体系
- 不做任务评论、日志和协作历史

## 5. 产品模型设计

### 5.1 用户与空间

- 每个用户最多属于 `1` 个双人空间
- 双人空间支持两种状态：
  - `pending`：已创建邀请，等待对方加入
  - `active`：双方绑定完成
- 未绑定期间，用户仍可正常使用个人清单和个人任务

### 5.2 清单模型

清单分为两类：

- `shared`：共享清单，归属于双人空间，两个人都可见、可编辑
- `private`：个人清单，只归属于清单拥有者本人，对方不可见

首版将“公开性”放在清单层，而不是任务层，原因是：

- 共享规则更稳定
- 首页筛选更直接
- 权限判断更简单

### 5.3 任务指派模型

任务支持指派给：

- 我
- 对方
- 双方

在数据结构上使用 `assigneeUserIds` 表达，而不是写死枚举值，便于后续渲染和判断。

### 5.4 个人先用，后续绑定

这是首版的重要前提：

- 用户无需先绑定双人空间才能开始使用
- 后续绑定成功后，个人数据仍保留
- 共享数据和个人数据并行存在

## 6. 数据模型设计

### 6.1 users

集合：`users`

建议字段：

- `userId`
- `openid`
- `nickname`
- `avatarUrl`
- `currentSpaceId`
- `createdAt`
- `updatedAt`

### 6.2 spaces

集合：`spaces`

建议字段：

- `spaceId`
- `status`
- `ownerUserId`
- `partnerUserId`
- `createdAt`
- `updatedAt`

### 6.3 spaceInvites

集合：`spaceInvites`

建议字段：

- `inviteId`
- `spaceId`
- `inviterUserId`
- `status`
- `expiresAt`
- `createdAt`
- `updatedAt`

### 6.4 lists

集合：`lists`

建议字段：

- `listId`
- `name`
- `type`（`shared` / `private`）
- `ownerUserId`
- `spaceId`
- `color`
- `sortOrder`
- `createdAt`
- `updatedAt`

说明：

- `shared` 清单必须带 `spaceId`
- `private` 清单的 `ownerUserId` 必须为当前用户

### 6.5 tasks

集合：`tasks`

建议字段：

- `taskId`
- `listId`
- `title`
- `dueDateKey`
- `isCompleted`
- `assigneeUserIds`
- `createdBy`
- `updatedBy`
- `createdAt`
- `updatedAt`

### 6.6 日期规则

- `dueDateKey` 固定保存为 `YYYY-MM-DD`
- 月视图查询、分组和展示均以 `dueDateKey` 为准
- 首版不处理具体时间点，只处理自然日
- 这样可以降低时区和自然日边界问题带来的复杂度

## 7. 关键业务规则

### 7.1 空间绑定规则

- 一个用户不能加入多个空间
- 若邀请已失效，则不能继续加入
- 若被邀请人已在别的空间中，则加入失败
- 若发起人对应空间状态已变化，则邀请需重新生成

### 7.2 可见性规则

- 共享清单：两个人都可见
- 个人清单：仅拥有者可见
- 首页默认“全部”不展示对方的个人清单内容

### 7.3 任务创建规则

- 未绑定双人空间时：
  - 只能使用自己的个人清单
  - 只能指派给自己
- 已绑定双人空间时：
  - 可以选择共享清单
  - 可以选择自己的个人清单
  - 不能把任务放进对方的个人清单

### 7.4 冲突规则

首版采用 `最后写入覆盖`。

原因：

- 产品目标是个人/双人轻协作，而不是复杂团队协作
- 可以显著压低 MVP 的同步成本
- 足以支撑首版核心场景

## 8. 系统架构

### 8.1 客户端技术路线

- 客户端：`原生微信小程序`
- 后端：`微信云开发`
- 数据库：云开发数据库
- 服务端逻辑：云函数

### 8.2 客户端分层

建议结构：

- `pages`
  - 首页
  - 任务编辑页
  - 清单管理页
  - 邀请处理页
- `components`
  - 月历头部
  - 月历网格
  - 日期格子
  - 当日任务列表
  - 任务行
- `stores`
  - 用户状态
  - 空间状态
  - 当前月份
  - 当前选中日期
  - 当前筛选范围
- `services`
  - 用户服务
  - 空间/邀请服务
  - 清单服务
  - 任务服务
  - 同步服务
- `utils`
  - 日期工具
  - 本地缓存
  - 权限判断

目标是：页面负责交互，服务层负责数据访问和同步，本地缓存与权限逻辑不要散落在页面内。

### 8.3 云函数职责

建议的核心云函数：

- `createSpaceInvite`
  - 创建邀请
  - 生成分享所需参数
- `acceptSpaceInvite`
  - 校验邀请有效性
  - 校验双方空间状态
  - 完成绑定
- `upsertTask`
  - 统一处理任务创建和编辑
  - 做权限校验
- `toggleTaskCompletion`
  - 统一处理勾选状态变更
- `manageList`
  - 统一处理清单创建和编辑

原则：

- 影响权限与共享关系的写操作优先走云函数
- 普通读取可直接查云数据库

## 9. 核心页面与交互

### 9.1 首页

核心职责：

- 展示任务在月份维度的分布
- 提供日期切换
- 展示当天任务详情
- 提供快速勾选和新增入口

### 9.2 任务编辑页

首版字段保持最小化：

- 标题
- 日期
- 所属清单
- 指派对象

新建任务默认继承当前选中日期。

### 9.3 清单管理页

首版只做轻量管理：

- 新建共享清单
- 新建个人清单
- 重命名清单

不做复杂排序、归档、颜色系统扩展。

### 9.4 邀请处理页

用于承接微信分享进入后的状态处理：

- 邀请有效且可加入：完成绑定
- 邀请失效：提示重发
- 已在其他空间：提示不可加入

## 10. 读取、同步与离线策略

### 10.1 首页读取策略

- 启动后先读取当前用户信息
- 再读取当前空间信息与可见清单
- 进入首页后按当前月份窗口加载任务
- 页面内按天聚合为月视图摘要与当天详情
- 切月份时只拉目标月份窗口

### 10.2 强离线策略

首版不做自研复杂同步引擎，但需要明确的本地缓存与待同步机制。

本地缓存内容：

- 最近访问月份的任务数据
- 当前用户信息
- 当前空间信息
- 清单信息

离线可执行操作：

- 查看已缓存月份
- 新增任务
- 编辑任务
- 勾选完成

离线写入方式：

- 本地先写缓存
- 同时写入待同步队列
- 网络恢复后按顺序重放到云端

### 10.3 冲突策略

- 首版默认最后写入覆盖
- 不做复杂冲突合并 UI
- 若后续场景明显升级，再单独设计冲突解决机制

## 11. 错误处理与边界

- 邀请失效：提示邀请无效或过期
- 用户已在其他空间：明确提示无法加入
- 空间已满：提示当前空间不可再加入
- 网络断开：使用本地缓存并保留待同步操作
- 同步失败：记录失败任务并允许后续重试

## 12. 测试策略

### 12.1 单元测试重点

- 日期与月份窗口计算
- 月视图聚合逻辑
- 可见性判断
- 指派对象判断
- 待同步队列处理

### 12.2 集成测试重点

- 新用户进入并创建个人任务
- 发起邀请并完成双人绑定
- 创建共享清单与共享任务
- 月视图切换与当天详情联动
- 离线新增/编辑/勾选，恢复网络后同步

## 13. 实现建议结论

基于已确认的产品边界，首版推荐路线为：

`原生微信小程序 + 微信云开发`

不推荐首版为了未来可能的跨端而先引入额外跨端框架。当前阶段最重要的是：

- 快速把微信生态内的登录、分享、绑定闭环跑通
- 把月视图主页体验做顺
- 把双人共享和强离线做成可信赖的 MVP

这条路线在当前目标下复杂度最低、成功率最高。
